#1 — Same-Identity Guard
─────────────────────────

What must change:
When any device presents its identity during session join, compare the
joining identity against the existing owner identity. If they match, do
NOT assign the joining device to the partner slot. Instead, treat it as
an additional owner device. This applies in both the invite-link join
flow and the WebSocket hello handshake.

What must be added:
A guard check at two locations: (a) in the invite admission evaluation
(evaluateInviteAdmission), and (b) in the hello handler
(handleRelayEvent, type==='hello'). At both points, before writing to
partnerIdentityId, check: if the incoming identityId ===
sess.ownerIdentityId, do not assign to partner. Instead, set localRole
to 'owner' and proceed as an owner-device-add.

What must be removed:
Nothing removed. This is purely additive.

Proof point:
Create a session on Device A. Open the partner-invite link on the same
device or a device with the same identity. The session must NOT show the
same identity in both ownerIdentityId and partnerIdentityId. Export
sessions JSON and confirm the two fields are never identical for any
session.

Codex instructions:
In function evaluateInviteAdmission(invite): after resolving sess, add:
if (invite.initiatorDeviceId === PART_ID || (sess.ownerIdentityId &&
sess.ownerIdentityId === PART_ID)) return {allowed:true,
redirectToOwner:true}. In the startup continueStartup() flow where
invite is processed, check if admission.redirectToOwner is true — if so,
call hydrateSessionFromInvite with the owner side, set
sess.localRole='owner', toast('Opening on this device as you'), and
proceed to openSession. In handleRelayEvent for type==='hello': in
upsertPartnerFromEnvelope, before writing d.identityId to
sess.partnerIdentityId, add guard: if (d.identityId ===
sess.ownerIdentityId) { addDiagLog('partner assign blocked — same as
owner identity','warn'); return; }


#2 — Upstream Redirect Before Join-Blocked Modal
─────────────────────────────────────────────────

What must change:
The existing join-blocked modal correctly handles genuine third-party
rejection. Upstream of it, when the joining identity matches the owner,
the user should be silently redirected into owner-device mode with a
toast, never seeing the blocked modal.

What must be added:
A check in the continueStartup() invite flow, after
evaluateInviteAdmission returns, that detects the redirectToOwner flag
from Item #1 and routes to openSession as owner instead of showing the
join-blocked modal.

Proof point:
Open a partner-invite link where your identity is already the owner. You
see a toast 'Opening on this device as you' and land in the session as
owner. You never see the join-blocked modal.

Codex instructions:
In continueStartup(), after var admission =
evaluateInviteAdmission(invite): add block: if
(admission.redirectToOwner) { sess = hydrateSessionFromInvite(sess,
Object.assign({}, invite, {side:'owner',
joinOrigin:'share-owner-handoff'})); sess.localRole = 'owner';
putSession(sess); toast('Opening on this device as you');
openSession(invite.sessionId); return; } This must appear BEFORE the if
(!admission.allowed) check.

#3 — Reduce Heartbeat Timeout to 45 Seconds
────────────────────────────────────────────

What must change:
The constant PING_TIMEOUT_MS currently set to 120000 (2 minutes). Change
to 45000 (45 seconds). Two strikes at 45-second intervals means dead
connections are detected in ~90 seconds instead of ~4 minutes.

Proof point:
After deploying, simulate a dead connection (e.g., airplane mode on
partner device). The diagnostic log should show HEARTBEAT timeout
strike=1 within 45 seconds, and reconnect within 90 seconds.

Codex instructions:
Change: var PING_TIMEOUT_MS = 120000; to var PING_TIMEOUT_MS = 45000;
— single constant change, no other logic affected.


#4 — Surface Silent Send Failures
──────────────────────────────────

What must change:
When 'reliable send blocked passive tab' occurs, the user currently sees
nothing. Add a visible toast and mark the affected message bubble.

What must be added:
(a) A toast message when sendReliable or transportSend returns false due
to passive tab: 'Message queued — switch to this tab to send'. (b) A
visual indicator on the bubble — add an amber-colored queued badge or
change the status dot to amber. (c) When the tab regains writer status
and the queue flushes successfully, update the bubble status back to
normal.

Proof point:
Open the app in two tabs. Send a message from the non-writer tab. You
see a toast saying the message is queued, and the bubble shows an amber
indicator. Switch to that tab (making it the writer). The message sends
and the indicator turns to normal sent status.

Codex instructions:
In sendMessage(), after var ok = sendReliable(payload): if (!ok) {
toast('Message queued — switch to this tab to send'); msg.status =
'queued'; } — the existing queued-badge CSS class .queued-badge already
exists. In updateMessageStatusUI, add handling for status==='queued'
that shows the badge in amber. In flushOutQueue, after successful send
of each item, find the corresponding message in state.messages and
update its status to 'sent', then call updateMessageStatusUI.

#5 — Per-Session Diagnostic Log
────────────────────────────────

What must change:
The diagnostic log (var diagLog=[]) is currently global. All sessions
share the same log buffer, so exports contain stale data from whichever
session was last active. Change to per-session log storage.

What must be added:
A diagLogByConv object keyed by convId. When addDiagLog is called,
append to both the global log and the active session's log. When
exporting a chat, include only that session's diagnostic entries. When
opening a session, set diagCurrentSess to that convId.

What must be removed:
Nothing removed. The global log remains for the diagnostics modal.
Per-session log is additive for exports.

Proof point:
Create two sessions. Send messages in each. Export both. Each export's
Diagnostics section contains ONLY log entries from that session
(identifiable by the session ID prefix in CONNECTED lines and message
IDs).

Codex instructions:
Add: var diagLogByConv = {}; Modify addDiagLog(msg, level): after
pushing to diagLog array, also push to
diagLogByConv[state.activeConvId || connectedConvId] (create array if
missing). In exportSessionChat(cid): replace the line that reads
diagLog.join('\n') with (diagLogByConv[cid] || []).join('\n'). Keep
MAX_DIAG=400 per session log as well. In openSession and
openInviteScreen, set diagCurrentSess = cid.

#6 — Validate Invite URL Base
──────────────────────────────

What must change:
Before generating any invite URL (both partner-invite and
owner-handoff), validate that location.href starts with https:// or
http://. If running inside an iframe (about:srcdoc), blob: context, or
file: protocol, do not generate the URL.

What must be added:
A validation check in buildInviteUrl. If the base URL fails validation,
return empty string. In renderInviteCard and showShareQr, when URL is
empty, show a message: 'Invite link unavailable — open this app directly
in a browser tab' instead of a blank QR.

Proof point:
Load the app in an iframe or srcdoc context. The invite card shows the
unavailability message instead of a blank QR box. No about:srcdoc URLs
appear in session exports.

Codex instructions:
In buildInviteUrl, before the return statement, add: var baseUrl =
location.href.split('?')[0].split('#')[0]; if
(!/^https?:\/\//i.test(baseUrl)) { addDiagLog('invite URL blocked:
invalid base ' + baseUrl.slice(0,30), 'warn'); return ''; } In
renderInviteCard, where url is checked: if (!url) show a div with text
'Invite link unavailable — open in a browser tab' styled with
color:var(--ink-dim), font-size:13px, text-align:center, padding:20px.
In showShareQr, if (!url) set qrState text to 'Share unavailable in
this context'.


#7 — Standardize QR Error Correction to Level M
────────────────────────────────────────────────

What must change:
The in-chat invite card uses QRCode.CorrectLevel.H (highest error
tolerance, lowest data capacity). The share overlay uses
QRCode.CorrectLevel.M. Standardize both to M so that longer URLs in
different repos don't silently fail.

Proof point:
Deploy to a repo with a longer base URL path. Both the partner QR
(invite card) and the owner QR (share overlay) render successfully.

Codex instructions:
In renderInviteCard, find the QRCode constructor call that uses
correctLevel:QRCode.CorrectLevel.H and change to
correctLevel:QRCode.CorrectLevel.M. The share overlay in showShareQr
already uses M — confirm and leave as-is.

#8 — One Immutable Announcement Per Event, Identical Both Sides
───────────────────────────────────────────────────────────────

What must change:
Currently, join/leave/name-change announcements can appear duplicated
(same person 'joined' twice at the same timestamp), appear
asymmetrically (owner sees partner joined but partner doesn't, or vice
versa), or appear in different order on each side. The announcement
system must produce exactly one system message per event, deduplicated
by eventId, and displayed identically on both sides.

What must be added:
(a) In projectSystemAnnouncements, before rendering, deduplicate by
eventId — if two canonical events have the same eventId, render only
one. (b) Sort announcements strictly by seq number, then by eventId as
tiebreaker. (c) The join event for a participant should be emitted
exactly once by the joining participant's device when the WebSocket
connects and the hello is acknowledged — not by each client
independently guessing that the peer joined.

What must be removed:
Remove any duplicate join-event emission from markPeerSeen.
markPeerSeen currently calls appendCanonicalEvent('join',...) when
peerPresent transitions from false to true. This causes both sides to
independently emit join events for the same participant. Instead, each
device should only emit its OWN join event (in resetActivePresence),
never the peer's.

Proof point:
Create a session, invite a partner. Both sides see exactly: '[Owner
name] joined · [time]' then '[Partner name] joined · [time]'. No
duplicates. Export both sides — the system messages are identical in
content and count. Reconnect — no spurious 'joined' announcements
appear.

Codex instructions:
1) In markPeerSeen: remove the block that calls
appendCanonicalEvent(convId,'join',...) for the peer. markPeerSeen
should only update presence state (rt.peerPresent, rt.peerLastSeenAt,
etc), never emit canonical events for the peer. 2) In
resetActivePresence: this already emits a join event for SELF — keep
this as the sole join-event source. 3) In projectSystemAnnouncements:
after collecting evs array, dedupe: var seen = {}; evs =
evs.filter(function(ev){if(seen[ev.eventId])return false;
seen[ev.eventId]=true; return true;}); 4) Ensure sort is strictly by seq
then eventId: evs.sort(function(a,b){if(a.seq!==b.seq)return
a.seq-b.seq; return
String(a.eventId).localeCompare(String(b.eventId));}); 5) In
handleRelayEvent for type==='hello': remove any
appendCanonicalEvent('join') call for the peer — the peer's own device
is responsible for emitting its join event via its own
resetActivePresence.

#9 — Delivery = Blue Dot, Read = Blue Checkmark
────────────────────────────────────────────────

What must change:
Replace the current single-dot status indicator with two distinct
states: a blue dot for delivered, and a blue checkmark for read. A
message is 'read' ONLY when all of: (a) the recipient's browser window
is in focus, (b) the tab is in focus, (c) the session card for that
conversation is the active session. If the session is already active and
the tab regains focus, move cursor to the composer input and mark all
previous unread messages as read at that moment.

What must be added:
(a) New status rendering: delivered shows a blue filled circle (Unicode
U+25CF) in color #2E8B8B at 10px. Read shows a blue checkmark (Unicode
U+2713) in color #2E8B8B at 12px. Undelivered shows a grey filled
circle in var(--ink-dim) at 10px. (b) A visibility/focus listener that,
on regaining focus with an active session, calls a new function
markSessionRead(convId) which iterates unread messages, stamps readAt,
updates UI, and sends read-receipt events to peer. (c) The composer
input auto-focuses when the session becomes the active session AND the
tab is focused.

What must be removed:
Remove the current logic that stamps readAt immediately upon message
receipt in handleRelayEvent (the line:
if(!existing.readAt){existing.readAt=Date.now();}). Read must only be
set by the focus-based mechanism.

Proof point:
Send a message from Device A. Device B receives it — Device A shows a
blue dot. Device B user focuses the tab and the correct session — Device
A's dot changes to a blue checkmark. If Device B is in a different
session or tab is unfocused, Device A stays on blue dot indefinitely.

Codex instructions:
1) In updateMessageStatusUI: replace dot rendering. Delivered = blue
filled circle U+25CF in color #2E8B8B at 10px. Read = blue checkmark
U+2713 in color #2E8B8B at 12px. Undelivered = grey filled circle in
var(--ink-dim) at 10px. 2) In handleRelayEvent type===msg: REMOVE the
line that sets existing.readAt or msg.readAt on receipt. Only set
deliveredAt. 3) Add function markSessionRead(convId): iterate
state.messages, for each msg where from===them and !msg.readAt, set
msg.readAt=Date.now(), call updateMessageStatusUI(msg), and send
read-receipt via transportSend with type:ack, ackType:read, id:msg.id.
Save messages after loop. 4) Add focus listener on visibilitychange: if
!document.hidden and state.activeConvId, call markSessionRead and focus
msg-input. Also in window focus handler, same logic. 5) In openSession:
after renderAllMessages, if !document.hidden call
markSessionRead(cid) and focus composer. 6) In handleRelayEvent for
type===ack where ackType===read: call
markMessageDeliveryState(convId, d.id, read, d.timestamp).


#10 — Unread Badge on Session Cards
────────────────────────────────────

What must change:
Each session card in the left panel must show a count badge of unread
messages — messages from the partner received since the last time that
session was marked as read.

What must be added:
(a) A function getUnreadCount(convId) that counts messages where
from==='them' and !msg.readAt. (b) In renderSessionList, for each
session card, if unreadCount > 0, render the existing .sess-card-badge
element with the count. (c) When markSessionRead runs, re-render the
session list to clear the badge.

Proof point:
Receive 3 messages while viewing a different session. The session card
for the receiving session shows a badge with '3'. Tap the card. Badge
disappears.

Codex instructions:
Add function getUnreadCount(convId) that returns
getMsgs(convId).filter(m => m.from==='them' && !m.readAt).length. In
renderSessionList, in the sess-card HTML generation, after the
sess-card-sub div: compute var unread = getUnreadCount(s.convId);
generate statusBadge HTML using the existing .sess-card-badge class with
the unread count, or empty string if zero. Insert statusBadge into the
card HTML before the action buttons div. In markSessionRead, after
saving messages, call renderSessionList() to refresh badges.
